## Build & release commands

```bash
# Lint
ruff check zettabrain_rag/

# Build the package (syncs setup.sh from scripts/ to root, then cleans and builds)
make build

# Bump version in pyproject.toml and __init__.py
make bump v=0.2.4

# Full release: bump + build + git commit + push + PyPI upload
make release v=0.2.4

# Upload to PyPI (after build)
make publish
```

There is no test suite. CI (`.github/workflows/ci.yml`) runs `ruff check` and `python -m build` only.

## Architecture overview

This is a Python package (`zettabrain-rag`) that wraps a fully local RAG pipeline: **LangChain + Ollama + ChromaDB**, with no cloud dependencies.

### Entry points â†’ CLI wiring

`pyproject.toml` registers CLI commands (`zettabrain-chat`, `zettabrain-ingest`, `zettabrain-setup`, `zettabrain-server`, `zettabrain-status`, `zettabrain-storage`) as entry points that all land in `zettabrain_rag/cli.py`. Each function there parses args and delegates to a Python script or shell script.

### Script deployment model

The core logic lives in `zettabrain_rag/scripts/` (bundled with the package). On first run, `cli.py:_deploy_scripts()` copies them to `/opt/zettabrain/src/`. At runtime, scripts are resolved from `/opt/zettabrain/src/` first, falling back to the package directory. This means installed scripts and the source package can diverge â€” edits to `zettabrain_rag/scripts/` only take effect after `pipx reinstall` or manual copy.

**Key scripts:**
- `03_langchain_rag.py` â€” interactive CLI chat (`zettabrain-chat`)
- `05_ingest_documents.py` â€” document ingestion with MD5 hash-based incremental tracking
- `01_chromadb_setup.py` / `02_embeddings_test.py` â€” diagnostics

### Configuration priority

All scripts share the same config resolution order:
1. Environment variable (e.g. `ZETTABRAIN_DOCS`)
2. `/opt/zettabrain/src/zettabrain.env` file (written by `setup.sh`)
3. Hardcoded fallback

Key config values: `ZETTABRAIN_DOCS`, `ZETTABRAIN_CHROMA`, `ZETTABRAIN_LLM_MODEL`, `ZETTABRAIN_EMBED_MODEL`, `ZETTABRAIN_CHUNK_SIZE`, `ZETTABRAIN_CHUNK_OVERLAP`, `OLLAMA_HOST`.

### Web server (`zettabrain-server`)

`zettabrain_rag/server.py` is a FastAPI app served by uvicorn. It exposes:
- `GET /api/status` â€” Ollama, vector store, and storage info
- `POST /api/chat` â€” single-turn RAG query
- `WS /ws/chat` â€” streaming chat via WebSocket (used by the web UI)
- `POST /api/ingest` â€” triggers ingestion subprocess
- `DELETE /api/vectorstore` â€” clears ChromaDB collection

The web UI is `zettabrain_rag/static/index.html` (served at `/`). TLS is optional; certs live at `/opt/zettabrain/certs/` and are generated by `setup.sh`.

### Vector store

ChromaDB persistent store at `/opt/zettabrain/src/zettabrain_vectorstore`, collection name `zettabrain_docs`. Ingestion tracks processed files via MD5 hashes in `ingested_files.json`. Retrieval uses MMR (Maximum Marginal Relevance) with `k=6, fetch_k=20, lambda_mult=0.7`.

### setup.sh / storage

`zettabrain_rag/scripts/setup.sh` is the canonical source. `make sync` (run automatically by `make build`) copies it to the repo root `setup.sh`. Never edit the root `setup.sh` directly.

## Version bumping

Version is stored in two places and must be kept in sync:
- `pyproject.toml`: `version = "X.Y.Z"`
- `zettabrain_rag/__init__.py`: `__version__ = "X.Y.Z"`

Use `make bump v=X.Y.Z` to update both atomically.
